# The Second Valley of Death — transition & profit-realization playbook

The technology has already won the arguments it can win: it performed, it piloted, warfighters validated it. The instinct to reach for more demos, more champions, more relationship-building answers a question the Department already answered. This is not a **proof** problem. It is a **programming** problem.

The gap between a successful pilot and a fielded program is the **second valley of death**. Cross it by engineering the bureaucracy with the same rigor we engineer the product. It has four structural walls.

## The four walls

**1. Money.** Pilots run on flexible RDT&E / OTA / innovation-cell dollars. Scaling requires *procurement* funds programmed into the POM roughly **two years before execution**. An enthusiastic program manager cannot buy what was never programmed.
- Find the **resource sponsor** (owns the money) — not just the end user (loves the capability). Get into the POM conversation *early*, before the demo.
- Bridge the gap with mechanisms purpose-built for this valley: **APFIT**, **mid-tier acquisition (MTA)**, **software acquisition pathway (SWP)**.

**2. Requirements.** Without a *validated requirement*, nobody has the authority to scale, no matter how good the demo was. Tie the capability to a documented gap / sponsor requirement so the authority to scale exists.

**3. Contracts.** If the pilot vehicle had no production path built in, success leads back to a full **recompete**, and momentum dies in source selection.
- Begin with the transition in mind: structure pilots on vehicles with a built-in scale path — **OTAs with follow-on production authority under 10 U.S.C. 4022(f)**, or **SBIR Phase III sole-source eligibility** — so success converts directly into a contract instead of a restart.

**4. Incentives.** A PM who integrates new tech owns every schedule slip it causes and captures none of the upside. That risk aversion is **rational**, not cultural. Make "yes" cheap and career-safe:
- **Open interfaces / MOSA compliance** — extend the baseline instead of threatening it.
- **Government Purpose Rights (GPR), in writing** — so IP/data negotiations never stall adoption.
- **ATO reciprocity** — so each new program isn't a fresh accreditation fight.

## The plays

1. **Begin with the transition in mind.** Pick the pilot vehicle for its scale path (4022(f) follow-on production, SBIR Phase III), not just for ease of award.
2. **Find the resource sponsor before the demo.** The end user opens the door; the resource sponsor owns the money. Get into the POM early.
3. **Cut the integration tax structurally.** MOSA + GPR-in-writing + ATO reciprocity make the PM's "yes" cheap and career-safe.
4. **Borrow transition infrastructure.** Ride proven rails — companies/vehicles with established ATOs, standing program-office relationships, existing contract vehicles — instead of building a pipeline from zero.

## Proof it works

Modern Day Marine 2026 AI Forum hackathon: an all-Marine team took **first place with SPIRE** (contested-logistics decision support). Before celebrating, they confirmed **Government Purpose Rights in writing** — so IP could never become the reason a program office said no. The demo opens the door; the paperwork lets the capability walk through.

## How to use this in the workspace

Each pursuit is scored on the four walls (Money / Requirements / Contracts / Incentives). The weakest wall is the next thing to engineer — not the next demo. The lifecycle runs bid → award → pilot → **transition** → POM → program of record → revenue, and the goal of every interaction is to make adoption the path of least resistance.
